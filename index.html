The Problem · Documented by GAO
FSA shipped a broken system — then stopped tracking whether it was fixed.
🔴 FAFSA Processing System
  • 9 of 25 requirements never deployed
  • 30% of forms had processing errors in 2024
  • SAI formula omitted asset data — 5M wrong aid offers
  • By May 2025 FSA stopped tracking the missing requirements entirely
🔴 Quality & Oversight
  • Test cases had no traceability to requirements
  • No user testing plan for future updates
  • Contractor performance never independently validated
  • Key staff lacked certifications for the methodology used
"FSA was unable to provide the status of the complete system... they were no longer tracking requirements." — GAO-25-107396, September 2025
These are documented failures, not hypothetical risks. The 2025 report found FSA couldn't tell auditors which requirements had been deployed.

No parallelism

Every step blocks the next. IRS and SSA could run simultaneously — they don't. Failures cascade silently.

No observability

IRS, SSA, and DHS are black boxes. Engineers only discover failures when students complain.

No feedback loop

Students and FAAs are equally blind. No status, no ETA, no explanation when something breaks.

This is the architectural root cause — not a UI problem, not a policy problem. Batch processing for what should be a real-time event-driven system.

The architectural shift: FAFSA submission becomes an event, not a batch job trigger. Agents run in parallel. Human approves the output. This is how modern fintech handles similar flows at this scale.
Agent Architecture
Each federal integration becomes an MCP server with a defined contract.
🤖 Orchestrator Agent

FAFSA event → parallel dispatch → collect results → assemble ISIR → route to FAA with audit package.

PM owns: state machine, timeouts, retry logic, escalation paths

📋 Audit Agent

Input, output, timestamp, confidence — all logged immutably. FERPA + GAO defensibility. Non-negotiable.

🔔 Notification Agent

Real-time status to students and FAAs. Eliminates the 3.5M unacknowledged forms problem.

🏦 IRS MCP Server

In: SSN + consent_token · Out: FTI validated schema
SLA <2s · Fallback: manual review queue

🔍 SSA MCP Server

In: Name + DOB + SSN · Out: confidence 0–1
SLA <1s · ≥0.85 auto-approve

🏛️ DHS MCP Server

In: Citizenship + residency · Out: eligibility flags
SLA <3s · Edge cases → manual review

📊 Eligibility Calculation

In: Validated IRS + SSA · Out: SAI + Pell + package
Formula: versioned, tested, auditable

MCP gives each integration a standard envelope: defined inputs, validated outputs, error codes, SLA requirements. The PM writes the contract spec. Engineers implement it. Auditors review it.

The Current State

What the GAO Found

Two GAO reports (2023, 2025) documented specific, preventable failures:

🔴 FAFSA Processing System (FPS)

  • Deployed Dec 2023 with 9 of 25 requirements undeployed
  • 30% of forms had processing/data errors (Mar–Apr 2024)
  • SAI formula missing asset data fields for dependent students
  • By May 2025: FSA stopped tracking whether those requirements were ever completed

🔴 Quality & Oversight Failures

  • Test cases had no traceability to requirements — impossible to verify if features worked as specified
  • No user testing plan for future FPS updates
  • Contractor performance data not validated
  • Key staff lacked required certifications for the dev methodology used
"FSA was unable to provide the status of the complete system... they were no longer tracking requirements." — GAO, Sept 2025
These aren't hypothetical risks. They're documented failures that affected real students. The technical PM role exists to prevent this from happening again.

Batch, not real-time

Every step waits for the previous step to complete. No parallelism. Any failure cascades silently.

Agent Architecture

The MCP-Connected Agent Layer

Each federal integration becomes a structured MCP server — well-defined inputs, validated outputs, full audit trail. The Technical PM owns the API contracts.

Orchestrator Agent

Receives FAFSA submission event → dispatches parallel agent calls → waits for all results → assembles ISIR → routes to FAA with audit package.

PM owns: State machine design, timeout handling, retry logic spec

Audit Agent

Every agent call — input, output, timestamp, confidence — logged immutably. Required for FERPA compliance and future GAO review.

PM owns: Audit schema, retention policy, access controls

Notification Agent

Real-time status to students and FAAs. Eliminates the "3.5M forms unacknowledged" problem. Configurable per institution.

PM owns: Message templates, opt-in/out spec, SLA thresholds

🏦 IRS MCP Server

Input: SSN + consent token
Output: FTI data object (validated schema)
SLA: <2s · Fallback: manual review queue

🔍 SSA MCP Server

Input: Name + DOB + SSN
Output: Identity confidence score (0–1)
SLA: <1s · Threshold: ≥0.85 auto-approve

🏛️ DHS MCP Server

Input: Citizenship/residency status
Output: Aid eligibility flags
SLA: <3s · Edge cases: manual review

📊 Eligibility Calculation

Input: Validated IRS + SSA output
Output: SAI + Pell eligibility + aid package
Formula: versioned, tested, auditable

The Technical PM doesn't write the agent code. They write the API contracts — the input/output schemas, the SLA requirements, the error handling specs — that engineers implement. That's the spec work this role owns.

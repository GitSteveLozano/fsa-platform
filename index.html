The Problem · Documented by GAO

FSA deployed a system in 2023 that broke aid for millions — and stopped tracking whether it was ever fixed.

FAFSA Processing System (FPS)

  • 9 of 25 contractual requirements never deployed
  • 30% of forms had processing errors Mar–Apr 2024
  • SAI formula omitted asset data — 5M incorrect aid offers
  • By May 2025, FSA had stopped tracking the missing requirements

Quality & Oversight Failures

  • Test cases had no traceability to requirements — no way to verify anything worked
  • No user testing plan for future updates
  • Contractor performance never independently validated
  • Key staff lacked certifications for the methodology used
"FSA was unable to provide the status of the complete system... they were no longer tracking requirements." — GAO-25-107396, September 2025
These aren't hypothetical risks. The GAO documented them. The 2025 report found that FSA couldn't even tell auditors which requirements had been deployed.

No parallelism

Every step blocks the next. IRS and SSA could run simultaneously — they don't. Any failure cascades silently.

No observability

IRS, SSA, and DHS are black boxes. Engineers only learn of failures when students report them.

No feedback loop

Students and FAAs are equally in the dark. No status, no ETA, no explanation when things break.

This is the architectural root cause. Not a UI problem, not a policy problem — a fundamental decision to use batch processing for a real-time problem. Everything else flows from this.

Human approval at the final step is legally required — not optional. The PM designs where humans stay in the loop.

IRS Income Agent

Real-time FTI via API. Validated before returning. Audit trail logged. No overnight batch. SLA: <2s

SSA Identity Agent

3-day wait → seconds. Runs parallel to IRS. Returns confidence score. Threshold-based approval. SLA: <1s

Eligibility Agent

SAI calculation on validated inputs. Versioned formula. Every input/output logged for FERPA. Auditable

The key architectural shift: event-driven instead of batch. FAFSA submission fires an event. Agents run in parallel. Humans approve the final output. This is achievable — it's how every modern fintech handles similar flows.
Agent Architecture

Each federal integration becomes an MCP server with a defined contract — the PM writes the spec, engineers implement it.

🤖 Orchestrator Agent

Receives FAFSA event → dispatches parallel calls → assembles ISIR → routes to FAA with full audit package.

PM owns: State machine, timeout handling, retry logic spec

📋 Audit Agent

Every call — input, output, timestamp, confidence — logged immutably. FERPA compliance + GAO defensibility.

🔔 Notification Agent

Real-time status to students + FAAs. Eliminates 3.5M unacknowledged forms. Configurable per institution.

🏦 IRS MCP Server

In: SSN + consent_token
Out: FTI (validated schema) · SLA <2s
Fallback: manual review queue

🔍 SSA MCP Server

In: Name + DOB + SSN
Out: Identity confidence 0–1 · SLA <1s
≥0.85 confidence → auto-approve

🏛️ DHS MCP Server

In: Citizenship + residency
Out: Aid eligibility flags · SLA <3s
Edge cases → manual review

📊 Eligibility Calculation

In: Validated IRS + SSA data
Out: SAI + Pell eligibility + package
Formula: versioned, tested, auditable

The MCP protocol gives each integration a standard envelope: structured inputs, validated outputs, error codes, SLA requirements. Engineers can build to this contract. Auditors can review it. Policy teams can verify compliance in the schema.
Technical PM Deliverable

  "endpoint": "POST /v1/irs/fti-verify",
  "input": {
    "ssn":           "string (encrypted)",
    "tax_year":      "integer",
    "consent_token": "expires 24h"
  },
  "output": {
    "agi":          "integer | null",
    "tax_status":   "filed|unfiled|nonfiler",
    "asset_value":  "integer | null",  ← missing in 2024

    "audit_id":     "uuid"
  },
  "sla":     { "p50": "800ms", "p99": "2s" },
  "fallback": "queue_manual_review"
}

Why this is the PM's job

  • Engineers need precise I/O specs to build without guessing
  • Policy teams verify compliance is encoded in the schema
  • The audit_id field satisfies FERPA without extra tooling
  • The fallback prevents cascading failure when IRS is down

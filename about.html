The Current State

What the GAO Found — Documented Failures, Not Opinion

Two GAO reports (GAO-23-106376 and GAO-25-107396) documented specific, preventable platform failures. These aren't risks — they already happened.

FAFSA Processing System (FPS)

  • Deployed Dec 2023 with 9 of 25 requirements undeployed
  • 30% of forms had processing/data errors (Mar–Apr 2024)
  • SAI formula missing asset_value for dependent students — 5M incorrect aid offers
  • By May 2025: FSA stopped tracking whether those 9 requirements were ever completed

Quality & Oversight Failures

  • Test cases had no traceability to requirements — impossible to verify what was actually built
  • No user testing plan for future FPS updates
  • Contractor performance data not validated by FSA
  • Key staff lacked certifications required for the development methodology used
"FSA was unable to provide the status of the complete system... they were no longer tracking requirements." — GAO-25-107396, September 2025

Why Batch Processing Is the Root Cause

The current architecture is sequential and opaque. Every step waits for the previous one, failures cascade silently, and there's no feedback loop to students or administrators:

Student submits FAFSA
  → Overnight batch processing (FPS queue)
  → IRS FTI request (async, days)
  → SSA identity verification (3-day wait)
  → SAI calculation (formula bugs)
  → ISIR batch sent to college (delayed)
  → FAA manually packages aid (weeks)

Timeline: 4–12 weeks
Result: 3.5M students unconfirmed for weeks, 30% with errors

This is not a UI problem. It's not a policy problem. It's a fundamental architectural decision — batch processing — that creates every downstream failure.

The Vision

From Weeks to Same-Day — API-First, Agent-Orchestrated

The same student aid process, redesigned around parallel agent calls instead of sequential batch jobs. Human approval is still required at the final step — by law, not just policy.

Student initiates FAFSA (event triggers)
  ↓
Orchestrator Agent dispatches parallel:
  ├─→ IRS Income Agent       (FTI lookup, <2s)
  ├─→ SSA Identity Agent     (verification, <1s)
  └─→ DHS Eligibility Agent  (residency check, <3s)
  ↓ (all run simultaneously)
Eligibility Calculation Agent
  ├─ Input: validated IRS + SSA + DHS results
  ├─ Output: SAI + Pell eligibility + aid package
  └─ Formula: versioned, tested, auditable
  ↓
Audit Agent logs everything (FERPA compliance)
  ↓
FAA Notification Agent alerts college in real-time
  ↓
FAA reviews and approves  ← human-in-the-loop required
  ↓
Student receives decision

Timeline: Same day

Orchestrator Agent

Receives FAFSA submission event. Dispatches parallel agent calls. Collects results. Assembles ISIR. Routes to FAA with full audit package.

PM owns: State machine design, timeout handling, retry logic spec, error escalation paths.

Audit Agent

Every agent call — input, output, timestamp, confidence score — logged immutably. Required for FERPA compliance and future GAO review. Non-negotiable.

🏦 IRS MCP Server

Input: SSN + consent token
Output: FTI (validated schema)
SLA: <2s · Fallback: manual review queue

🔍 SSA MCP Server

Input: Name + DOB + SSN
Output: Identity confidence (0–1)
SLA: <1s · Threshold: ≥0.85 auto-approve

API Contract Design — What a Technical PM Actually Produces

The Technical PM doesn't write agent code. They write the API contracts — precise input/output schemas, SLA requirements, error handling specs, and audit requirements — that engineers implement and auditors review.

Example: IRS FTI Verification Contract

{
  "agent": "IRSFTIAgent",
  "endpoint": "POST /v1/agents/irs-fti-verify",

  "input": {
    "ssn":           "string (encrypted in transit)",
    "tax_year":      "integer  // e.g. 2025",
    "consent_token": "string   // expires 24h after FAFSA submission"
  },

  "output": {
    "agi":          "integer | null     // Adjusted Gross Income (cents)",
    "tax_status":   "filed | unfiled | nonfiler",
    "asset_value":  "integer | null     // CRITICAL: required for accurate SAI",
    "confidence":   "float 0.0–1.0     // <0.8 = recommend manual review",
    "audit_id":     "uuid               // immutable compliance reference"
  },

  "sla": { "p50_ms": 800, "p99_ms": 2000 },

  "error_handling": {
    "consent_expired":  { "http": 401, "fallback": "route_to_manual_review" },
    "ssn_not_found":    { "http": 404, "fallback": "route_to_manual_review" },
    "irs_unavailable":  { "http": 503, "fallback": "retry_3x_then_manual",
                          "retry_after_hours": 1 }
  },

  "audit": {
    "all_calls_logged": true,
    "retention_years":  7,
    "ferpa_compliant":  true
  }
}

90-Day Plan

How I'd Approach the First 90 Days

The biggest risk is arriving with a plan and executing against the wrong problem. The first month is about earning the standing to influence, not demonstrating what I already know.

Day 1–30

Listen, Map, Don't Break Anything

Shadow every engineering team. Read every GAO finding. Map the architecture — what exists, what's broken, who owns what. Build relationships with FSA leadership, contractor PMs, and IRS/SSA/DHS integration owners. Audit the 9 undeployed FPS requirements and map their dependencies.

Day 31–60

Instrument and Prioritize

Stand up DORA metrics across all teams. Build the requirement traceability matrix (user need → spec → test → deployed). Present a prioritized tech debt backlog with student impact quantified. Get FSA leadership alignment on P0 items before proposing solutions.

Day 61–90

Ship One Thing That Matters

Own the API contract spec for one integration (IRS FTI or SSA). Coordinate with engineers, legal, and policy. Get it deployed and verified. Prove the traceability framework works: requirement → test → deployed → confirmed in production.

Year 1

Architecture for the Agentic Future

Propose the MCP-connected agent architecture for IRS/SSA/DHS integrations. Build internal alignment across FSA, contractors, and agencies. Pilot one integration as an API-first proof of concept. Demonstrate the path: batch → event-driven, weeks → same-day.
